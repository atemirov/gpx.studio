## 2. Что такое SASPlanet (кратко, для контекста)

Desktop-утилита (Windows, есть Linux через Wine) для просмотра, скачивания и **локального
кэширования** тайлов карт/спутниковых снимков из десятков источников (Google, Bing, Yandex,
DigitalGlobe, «Космоснимки», OSM и др., включая карты Луны и Марса). Открытый исходный код,
активно разрабатывается (последний релиз — апрель 2026). Позиционирование принципиально другое,
чем у нас: **desktop-инструмент для оффлайн-скачивания карт**, а не веб-редактор GPX-треков.
Прямых пересечений в архитектуре мало, но есть конкретные фичи, которые стоит перенять.

## 3. Функциональность по категориям

### 3.1 Оффлайн-кэш тайлов
Ядро программы: скачивает тайлы z1–z20 из выбранных источников, хранит локально (SQLite/файлы),
работает без сети. Есть «пересборка» тайлов между масштабами для экономии трафика.

**Релевантность: высокая.** У нас веб-SPA — полный аналог невозможен и не нужен (нет цели делать
desktop-качалку карт), но частичный **оффлайн-режим на базе Service Worker** (кэшировать
уже просмотренные тайлы для использования в поле без сети) прямо ложится на сценарий
эндуро/велопоходов из описания проекта. Это уже отдельный пункт бэклога («PMTiles в росс. S3»)
частично решает смежную задачу (устойчивость к сбоям источника тайлов), но не оффлайн-режим для
конечного пользователя в поле.
**Сложность: средняя-высокая** (Service Worker + Cache API, лимиты браузерного стораджа,
UI для управления кэшем/выбора зоны для оффлайн-скачивания). Отдельная фича, не тривиальная
надстройка.

### 3.2 Экспорт форматов
Из вики и changelog релизов: экспорт в OziExplorer, iPhone/Yandex Maps, **JNX (Garmin)**,
RMaps SQLite (мобильные платформы), а из недавних релизов — **MBTiles, GeoTIFF, WEBP,
TIFF-DEM, KMZ/KML**, растущая поддержка GeoJSON.

**Релевантность: средняя, точечно высокая.**
- Экспорт в **Garmin (FIT/GPX уже есть в апстриме, JNX — нет)** — пересекается с уже
  зафиксированным пунктом бэклога «Экспорт в форматы навигаторов (Garmin FIT)» в Фазе 6.
  JNX — узкоспециализированный растровый формат Garmin для карт (не треков), не относится к
  экспорту треков; отдельная фича низкого приоритета, если вообще нужна (у нас нет своих
  растровых карт для экспорта).
- KML/KMZ — стоит проверить, есть ли уже в апстриме gpx.studio (велика вероятность, что да,
  т.к. GPX↔KML — стандартная пара форматов для трек-редакторов).
- MBTiles/GeoTIFF — релевантны, только если возьмёмся за оффлайн-кэш (см. 3.1): MBTiles —
  разумный формат для «скачать этот регион офлайн» фичи.
**Сложность:** JNX — высокая (сложный бинарный формат, малая отдача); KML — низкая, если ещё не
реализовано (стоит сначала проверить `gpx/` библиотеку); MBTiles — средняя, только в связке с 3.1.

### 3.3 Кастомные источники карт (формат ZMP)
SASPlanet использует расширяемый формат «карта в zip» (`Params.txt`, `GetUrlScript.txt`,
URL-шаблоны, коды проекций EPSG) — позволяет сообществу добавлять произвольные тайловые
источники без изменения кода программы. Отсюда — большая экосистема карт (Генштаб, топокарты
и т.п.), распространяемых через git-репозиторий с авто-обновлением.

**Релевантность: низкая для нас напрямую**, но идея показательна: у нас источники слоёв уже
конфигурируемы через `website/src/lib/assets/layers.ts` + `.env`-переменные — то есть тот же
принцип «источник карты — это конфиг, не код» у нас уже реализован, просто без
пользовательского UI для добавления своих источников. Полноценный ZMP-аналог (пользователь сам
вставляет URL-шаблон тайлового сервера через UI) — интересная, но некритичная фича для будущего.
**Сложность: средняя** (нужны валидация URL-шаблона, проверка CORS на лету, UI формы) —
не приоритет сейчас.

### 3.4 GPS/навигационные функции
Приём координат с GPS-приёмников по NMEA (COM/USB/Bluetooth) — SASPlanet работает как навигатор
в реальном времени поверх скачанных карт. Также: прокладка маршрутов, замер дистанций,
элевационные профили маршрута (добавлены в недавних релизах), отображение доп. данных при
наведении курсора.

**Релевантность: низкая.** Приём live-GPS с устройства — вне сценария нашего продукта (веб-SPA
для планирования/редактирования треков, не полевой навигатор в реальном времени; Web
Geolocation API технически мог бы дать позицию телефона в браузере, но это отдельная,
некоторыми пользователями не запрашивавшаяся фича). Замер дистанций и элевационный профиль **у
нас уже есть** (Chart.js elevation profile — родная фича gpx.studio). Ничего нового отсюда
брать не нужно.

### 3.5 DEM/высоты и оффлайн-роутинг
Документация sasgis.github.io упоминает: получение высоты под курсором из DEM-слоя,
экспериментальный оффлайн-роутинг, чтение кэшей сторонних программ (Google Earth, GeoCacher).

**Релевантность: низкая-средняя.**
- Высота под курсором — у нас уже есть эквивалент через профиль высоты трека
  (`getElevation()` в `website/src/lib/utils.ts`, на базе Mapterhorn DEM) — точечный запрос
  высоты под курсором мыши (не только по треку) можно рассмотреть как мелкое UX-улучшение,
  но это не задача SASPlanet-масштаба.
- Оффлайн-роутинг — у нас роутинг уже self-hosted (BRouter), реальный оффлайн (в браузере)
  потребовал бы портировать движок роутинга в WASM — несоразмерно сложно относительно пользы.
**Сложность:** высота под курсором — низкая (если решим делать); оффлайн-роутинг — очень
высокая, не рекомендуется.

### 3.6 Слой спутниковых снимков / карта покрытия
Множество спутниковых источников, карта «что где заснято и с каким качеством» (coverage map).

**Релевантность: уже закрыто.** У нас есть `esriSatellite` в базовых картах (унаследовано из
апстрима, см. `website/src/lib/assets/layers.ts`) — отдельного действия не требуется, уже
отмечено в Фазе 6 ранее.

## 4. Форум/сообщество — сигнал по эндуро/бездорожью

Целевой веб-поиск («SASPlanet карты эндуро бездорожье грунтовки форум лесные дороги») предметного
сигнала не дал — тематические ветки форума посвящены обмену конкретными картами (топокарты,
источники снимков), а не запросам фич под эндуро-сценарии. FAQ форума (viewtopic.php?f=2&t=986)
подтверждает фокус программы на скачивании/кэшировании карт и NMEA-GPS-приём, явно **не**
упоминает: DEM/высоты, роутинг, редактирование треков — то есть трек-редактирование в принципе
не входит в зону ответственности SASPlanet, это ожидаемо при их позиционировании как
«карт-качалки», а не трек-редактора. Дальнейшее погружение в форум не даст нового сигнала для
наших целей — закрываем этот источник.

## 5. Итоговые рекомендации (приоритизировано)

| # | Идея | Релевантность | Сложность | Статус |
|---|---|---|---|---|
| 1 | Оффлайн-кэш тайлов (Service Worker) для использования в поле без сети | высокая | средняя-высокая | новая идея, в бэклог Фазы 6 |
| 2 | Экспорт в KML/KMZ (если ещё нет в апстриме — проверить) | средняя | низкая (если нет) | требует проверки `gpx/` перед оценкой |
| 3 | Пользовательский UI для добавления кастомных тайловых источников (аналог ZMP) | низкая | средняя | не приоритет |
| 4 | Высота под курсором мыши (не только по треку) | низкая-средняя | низкая | мелкое UX-улучшение, не приоритет |
| 5 | Экспорт в Garmin JNX | низкая | высокая | не рекомендуется — узкий формат, малая отдача |
| 6 | Live GPS/NMEA приём, оффлайн-роутинг в браузере | очень низкая | очень высокая | отклонено — вне сценария продукта |

**Главный практический вывод:** из всей функциональности SASPlanet по-настоящему ценна для нас
только идея **оффлайн-доступности** (пункт 1) — она резонирует с уже заявленной аудиторией
проекта (эндуро/велопоходы, поля без связи) и с уже существующим пунктом бэклога про устойчивость
тайлового источника (PMTiles в росс. S3). Остальные пункты SASPlanet либо уже закрыты в проекте
(спутниковый слой, элевационный профиль), либо не вписываются в позиционирование веб-редактора
(живой GPS-навигатор, десктопная карт-качалка), либо дают низкую отдачу относительно сложности
(JNX, оффлайн-роутинг).

**Рекомендация по следующему шагу:** не запускать реализацию сейчас (задача была только
discovery). Если пользователь решит продолжать — пункт 1 (оффлайн-кэш через Service Worker)
стоит оформить отдельной задачей с собственным дизайном (сколько данных кэшировать, как
пользователь выбирает зону/маршрут для оффлайн-скачивания, лимиты Cache API в разных браузерах).
