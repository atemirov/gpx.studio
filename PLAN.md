# План проекта: GPX-редактор на базе gpx.studio

Версия 1.0 · Июль 2026 · Реализация через агентов Claude Code

---

## 0. Резюме и стратегическое решение

**Решение: форк gpx.studio (MIT), а не реимплементация с нуля.**

Обоснование:
1. Цель — работающий продукт («точная копия» + свои фичи), а не учебное упражнение. Форк даёт 100% паритет функциональности в день один.
2. Лицензия MIT разрешает форк, переименование, коммерческое использование. Единственное обязательство — сохранить текст лицензии и копирайт.
3. Экономия 6–10 недель агентного времени, которое перенаправляется на дифференциаторы (слой публичных GPS-треков, self-hosted инфраструктура, русификация).
4. Оригинал активно развивается — форк позволяет периодически подтягивать upstream-фиксы.

**Что получаем на выходе:**
- Работающий GPX-редактор на собственном домене и VPS.
- Полная независимость от платных западных API (тайлы, роутинг, высоты) — всё self-hosted или на бесплатных открытых сервисах.
- Фича, которой нет в оригинале: слой общедоступных GPS-треков OSM (аналог `&layers=G` на openstreetmap.org).
- Русский язык как основной.

**Оценка:** 5 фаз, ~10–15 агентных сессий, 2–4 недели календарно при контроле в режиме part-time.

## 3. Правила работы агентов

1. **Одна задача — одна ветка — один агент.** Merge в `main` только через твоё ревью диффа.
2. **Тесты — контракт приёмки.** Агент не завершает задачу, пока `npm test` и `npm run build` зелёные. Для UI-задач — Playwright-смок (страница открывается, файл грузится, трек виден).
3. **CLAUDE.md — единственный источник соглашений.** Каждый агент читает его первым. Обновление CLAUDE.md — часть definition of done, если задача меняет структуру/команды.
4. **Не переписывать — адаптировать.** Мы форк: минимальный дифф к upstream, чтобы сохранить возможность merge'ить их обновления. Крупные новые фичи — в отдельных модулях/директориях.
5. **Параллельность:** независимые задачи внутри фазы можно гнать параллельно через git worktree + tmux (у тебя есть agent-manager-skill — он для этого и подходит). Задачи с общим кодом — строго последовательно.
6. **Язык:** код и идентификаторы — английский; коммиты, комментарии к PR, документация — русский.
7. **Все агенты работают на ноутбуке**, в клоне репозитория. Infra-агент готовит конфиги локально в `infra/` и применяет на VPS через SSH только после явного подтверждения человеком. Claude Code на боевом сервере не запускается.
8. **Права в Claude Code:** без подтверждения — `npm install/run/test`, `git` внутри рабочей ветки; ручное подтверждение — `ssh`, `rsync`, `git push`, любые команды к боевому серверу.
9. **Последовательность/параллельность:** Фазы 0–1 — один агент строго последовательно (формируют фундамент). С Фазы 2 независимые задачи параллелятся: git worktree на задачу + tmux-сессия на агента (agent-manager-skill).

### Фаза 2 — Инфраструктурная независимость

**Цель:** ноль платных/лимитированных западных API. Три независимые задачи — можно параллелить.

**Задача 2.1 — Тайлы карт.**
Целевое решение: базовый слой — OpenFreeMap (бесплатные векторные тайлы без ключа и лимитов). Страховка (активировать ТОЛЬКО при проблемах с OpenFreeMap из РФ): свои PMTiles в российском S3-совместимом хранилище (Yandex Object Storage / Selectel / VK Cloud / Timeweb) — PMTiles читается range-запросами напрямую из бакета, диск VPS не нужен. Требования к бакету: публичное чтение, поддержка Range, CORS (GET/HEAD + заголовок Range с нашего домена); перед бакетом — nginx proxy_cache на VPS для экономии исходящего трафика S3.
> Промпт: По docs/ARCHITECTURE.md найди конфигурацию tile-источников. Добавь OpenFreeMap (стиль liberty) как базовый слой по умолчанию, сохранив MapTiler-слои как опциональные (работают только при наличии ключа). Убедись, что 3D-рельеф деградирует корректно, если terrain-источник недоступен. Обнови меню слоёв. Тесты: приложение полностью работает с пустым .env.

**Задача 2.2 — Роутинг: BRouter на твоём VPS.**
BRouter лёгкий (Java, сегментные файлы рельефа по регионам, для РФ+СНГ порядка десятков ГБ), есть готовые docker-образы; у авторов gpx.studio есть свой форк brouter — можно взять их конфиг как референс.
> Промпт для infra-агента: Собери в infra/brouter docker-compose для BRouter с сегментами Восточной Европы и европейской части РФ, с nginx-реверсом и rate-limit. Напиши скрипт обновления сегментов по cron. Проверь HTTP API запросом маршрута Тверь→Торжок, приложи пример ответа.
> Промпт для frontend-агента: Найди в коде обращения к движку роутинга. Вынеси URL движка в конфиг (env). Реализуй адаптер к BRouter API (формат ответа отличается от GraphHopper — сверься с документацией BRouter). Профили: авто, вело, пешком + добавь профиль для эндуро на базе вело-профиля с разрешёнными грунтовками. Тест: прокладка маршрута через наш BRouter работает во всех профилях.

**Задача 2.3 — Высоты точек.**
> Промпт: Найди, откуда приложение берёт высоту для новых точек (terrain-тайлы или API). Сделай источник конфигурируемым; primary — данные, не требующие ключа (terrarium-тайлы AWS Open Data или self-hosted DEM из Copernicus GLO-30), fallback — отсутствие высоты без падения UI.

**Критерии приёмки фазы:** приложение с пустым `.env` (без единого ключа) полностью функционально: карта, роутинг, высоты.

### Фаза 4 — Слой публичных GPS-треков OSM (ключевой дифференциатор)

**Уровень 1 — растровый оверлей (MVP, обязательный).**
> Промпт: Добавь в меню слоёв секцию «Оверлеи» → «GPS-треки OSM». Источник: raster-тайлы https://gps.tile.openstreetmap.org/lines/{z}/{x}/{y}.png, tileSize 256, атрибуция «© OpenStreetMap contributors». Слой должен: включаться/выключаться чекбоксом, сохраняться в состоянии слоёв и в URL-hash (по аналогии с тем, как это делает openstreetmap.org через &layers=G), корректно ложиться ПОД слои пользовательских треков, но НАД базовой картой. Добавь в конфиг возможность переопределить URL тайл-сервера (на случай self-hosted зеркала). Playwright-тест: включение слоя добавляет raster-layer в стиль карты.

**Уровень 2 — векторные треки через OSM API (после MVP).**
> Промпт: Создай модуль website/src/lib/osm-traces/: клиент к OSM API 0.6 GET /api/0.6/trackpoints?bbox=&page= (bbox ≤ 0.25 кв. градуса, ≤5000 точек/страница, пагинация до исчерпания). Требования: слой активен только на зуме ≥13; при движении карты — debounce 500 мс; кэш ответов по тайловой сетке bbox'ов (LRU в памяти + Dexie); отрисовка GeoJSON-линиями с настраиваемыми цветом/толщиной/прозрачностью в панели слоя; кнопка «использовать как референс» — фиксирует текущие треки, чтобы они не перезагружались при движении карты. Обработка ошибок и rate-limit OSM API — обязательна (экспоненциальный backoff).

**Критерии приёмки:** уровень 1 — визуальный паритет с openstreetmap.org при layers=G; уровень 2 — на зуме 14 в районе с треками линии появляются ≤3 сек, карта не фризится.
